# PRD 작성 Skill

## 목적
난잡한 기획 메모·회의 내용을 사내 표준 PRD 양식으로 변환한다.
PRD는 "무엇을·왜" 만드는지 제품 관점에서 정의하는 문서이며, 여기서 정한 요구사항이
`SRS`(무엇을 검증 가능한 조건으로)·`SDD`(어떻게 구현하는지)의 상위 근거가 된다.

## 작성 절차
1. 입력(메모/회의록/기획서)에서 배경·목적·목표·사용자·기능/비기능 항목을 추출한다.
2. `references/prd-template.md` 의 표준 양식을 기준 골격으로 삼는다. 각 섹션의
   `<!-- -->` 주석 안내를 따라 채우고, 채운 뒤에는 해당 주석을 지운다.
3. 각 요구사항에 표 형태로 고유 ID를 부여한다.
   - 사용자 스토리: `US-01`
   - 기능 요구사항: `F-01`
   - 비기능 요구사항: `NF-01`
4. 각 요구사항은 "측정 가능한 조건"으로 쓴다.
   - 나쁨: "빠르게 응답한다"
   - 좋음: "100페이지 이하 PDF 1개는 10초 이내에 검사를 완료한다(NF-01)"
5. 정보가 비어 있으면 추측하지 말고 `[확인 필요: …]` 로 표시한다.

### 구체적 작성 팁
- 결론 먼저 — 의사결정자가 30초 안에 파악하게
- 1요구사항 = 1행(표), 테스트 가능하게
- 도메인 특화 섹션(설정/데이터 구성 등)은 4장과 5장 사이에 자유롭게 추가
- 살아있는 문서 — 결정 바뀌면 갱신. 하위 문서(SRS/SDD)와 ID·용어를 맞춘다

## 출력 형식
파일 맨 위 골격은 `references/prd-template.md` 와 동일한 H2 구조를 따른다.
산출물은 `.md` 로 작성한다.

## 발행 전 자가 점검
초안을 끝낸 뒤, 반드시 아래 검증을 통과시킨 다음 사용자에게 제시한다:
`python ${CLAUDE_SKILL_DIR}/scripts/validate.py <작성한_파일경로>`
실패 항목이 있으면 고친 뒤 다시 돌린다. 통과 후에만 최종본을 제시한다.

## 발행 전 판단 점검
스크립트 통과 후, 아래를 스스로 자문하고 미흡하면 수정한다:
- [ ] 각 요구사항이 정말 "테스트로 검증 가능"한가? (참/거짓을 가릴 수 있는가)
- [ ] 기능/비기능 요구사항이 서로 모순되지 않는가?
- [ ] 범위 외(마지막 장)에 해당하는 기능이 4장에 슬쩍 들어가지 않았는가?
- [ ] [확인 필요] 항목을 사용자에게 질문 목록으로 따로 정리했는가?
- [ ] 기존 SRS/SDD가 있다면 ID·용어가 이번 변경과 어긋나지 않는가?
