# SRS 작성 Skill

## 목적
난잡한 기획 메모·회의 내용을 사내 표준 SRS 양식으로 변환한다.
요구사항이 검증 가능하고(verifiable) 측정 가능하도록(measurable) 다듬는 것이 핵심이다.

## 작성 절차
1. 입력(메모/회의록/기획서)에서 목적·범위·이해관계자·기능/비기능 항목을 추출한다.
2. `references/srs-template.md` 의 표준 양식을 기준 골격으로 삼는다. 각 섹션의 `<!-- -->` 주석 안내를 따라 채우고, 채운 뒤에는 해당 주석을 지운다.
3. 각 요구사항에 고유 ID를 부여한다. 정의는 기능 `[FR-001]`·비기능 `[NFR-001]`처럼
   대괄호로 한 줄에 단독 표기하고, 요구사항 내용은 그 아랫줄부터 쓴다.
   다른 문장에서 그 요구사항을 가리킬 땐 대괄호 없이 `FR-001`로 참조한다.
4. 각 요구사항은 "측정 가능한 조건"으로 쓴다.
   - 나쁨: "빠르게 응답한다"
   - 좋음:
     ```
     [NFR-003]
     검색 응답은 95% 요청에서 1초 이내로 반환한다.
     ```
5. 정보가 비어 있으면 추측하지 말고 `[확인 필요: …]` 로 표시한다.

## 출력 형식
파일 맨 위 골격은 `references/srs-template.md` 와 동일한 H2 구조를 따른다.
산출물은 `.md` 로 작성한다.

## 발행 전 자가 점검
초안을 끝낸 뒤, 반드시 아래 검증을 통과시킨 다음 사용자에게 제시한다:
`python ${CLAUDE_SKILL_DIR}/scripts/validate.py <작성한_파일경로>`
실패 항목이 있으면 고친 뒤 다시 돌린다. 통과 후에만 최종본을 제시한다.

## 발행 전 판단 점검
스크립트 통과 후, 아래를 스스로 자문하고 미흡하면 수정한다:
- [ ] 각 요구사항이 정말 "테스트로 검증 가능"한가? (참/거짓을 가릴 수 있는가)
- [ ] 기능/비기능 요구사항이 서로 모순되지 않는가?
- [ ] 범위(2장)에 없는 기능이 6장에 슬쩍 들어가지 않았는가?
- [ ] [확인 필요] 항목을 사용자에게 질문 목록으로 따로 정리했는가?
